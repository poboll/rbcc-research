# 产品使用指南

本指南按一次真实调研的顺序组织。不要从报告页开始补结论；先建立问题与证据，再让痛点、方案和报告自然生长。

## 1. 队员端

进入 `/app`，选择成员身份、日期、线路和站点。队员可以维护预设问题、填写验证状态，并上传图片、录音和文字留痕。

验证结论应区分：

- 成立：已有明确现场依据。
- 部分成立：只验证了假设的一部分。
- 推翻：现场信息与原假设相反。
- 待验证：缺少足够证据，不应为了提高比例强行判断。

推荐操作顺序：选择本人 -> 选择日期和站点 -> 阅读预设问题 -> 采集留痕 -> 回到问题填写结论与证据来源。保存后刷新一次，确认数据已持久化。

## 2. 留痕与知识

`/traces` 用于按成员、日期、站点和类型筛选现场证据；`/knowledge` 用于挂载团队资料并建立可检索片段。

留痕标题说明“看到了什么”，正文说明“它能支持什么”，不要直接把推断写成事实。知识文档用于提供背景和方法，不应覆盖现场记录。

## 3. 红八宝 Agent

`/agent` 优先检索本组资料。输出中的“本组事实”可以进入报告；“通识补充”和“待验证”必须继续寻找证据，不能直接当作企业结论。

## 4. 痛点与方案

协同 Hub 将问题和证据收敛为痛点，再为痛点建立最小试验方案。方案至少应包含目标用户、试验动作、衡量指标、风险、人工兜底和停止条件。

一个痛点最好关联多个问题或多个站点；只有单条弱证据时，先保留为观察点。方案不是宏大愿景，而是可在有限时间内验证的最小实验。

## 5. 报告与评审

`/dashboard` 按现状扫描、人群共情、痛点诊断、分析对策四核结构组织报告。管理员可上传 DOCX 定稿，教师通过 `/review` 查看代表站点、证据和下载入口。

## 6. 路演

`/screen/roadshow` 提供 26 幕电脑演示，支持播放、暂停、上一幕、下一幕、从头播放、空格键和方向键。演示读取真实数据，不会修改业务状态。

## 7. 管理端

管理员在 `/admin` 维护团队配置、证据、知识、痛点、方案、报告版本和 DOCX 定稿。执行删除、发布或批量修改前先导出备份。管理员凭证见[安全与管理员访问](Security-and-Admin)。

## 8. 汇报前检查

- 五名成员和路线归属正确。
- 关键结论能打开对应证据。
- 验证比例来自真实状态，不是手工展示数字。
- 痛点关联问题与证据，方案包含最小实验和停止条件。
- 教师评审能下载当前 DOCX 定稿。
- 路演在目标电脑和投影比例下完整播放。
